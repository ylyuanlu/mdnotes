# ADR-0012 — v2.0 架构升级 Proposal

> **日期**：2026-06-21
> **状态**：Proposed
> **目标版本**：v2.0.0
> **决策者**：PM Agent
> **相关文档**：ADR-0007（v1.0 复盘）/ ADR-0009（v1.5 复盘）/ `docs/PRD/01-cli-mvp.md` ~ `06-search-optimization.md`

---

## 背景

v1.5 阶段完成了功能补完（soft-delete + search 优化），是 v1.x 系列的收尾。ADR-0007 和 ADR-0009 均明确将 v2.0 定位为**架构升级**阶段，而不是下一个功能版本。

本 proposal 解决两个问题：
1. **v2.0 包含哪些架构维度**，优先级和风险如何
2. **用几个 minor 版本过渡**，还是直接大版本发布

---

## v2.0 动机：为什么是架构升级

| 对比维度 | v1.x（功能补完期） | v2.0（架构升级期） |
|---|---|---|
| 目标 | 补完 v1.0 设计的功能缺口 | 打破单 vault / 单机 / 单用户天花板 |
| 变更性质 | 增量功能（add/delete/search） | 结构性变更（存储模型 / 同步层 / API 层） |
| 影响范围 | CLI 命令参数兼容 | 数据库 schema / CLI 接口 / 可能有 breaking change |
| 典型任务周期 | 1~2 个 sprint | 3~5 个 sprint |
| 风险级别 | 低（原地加功能） | 中高（跨版本接口重构） |

**v1.5 复盘结论**（ADR-0009 §后续阶段建议）：

> v2.0（架构升级）
> - 多 vault 支持：`mdnotes --vault <name> add` 支持多个独立笔记仓库
> - 远程同步：加密上传到云存储（AWS S3 / Cloudflare R2）
> - Web UI：基于 FastAPI 的 Web 界面（长期路线）

**v1.0 复盘结论**（ADR-0007 §后续阶段建议）：

> v2.0：架构升级（多 vault / 远程同步 / Web UI）

两次复盘一致指向同一个方向：v2.0 是架构层面的扩展，不是功能层面的堆砌。

---

## 候选范围

### C1 — 多 vault 支持

**问题**：当前所有笔记存在单一 SQLite（`~/.mdnotes.db`），无法管理多个独立笔记仓库。用户想用 mdnotes 同时管"工作笔记"和"个人笔记"，目前只能开两个目录跑两份 mdnotes。

**目标**：一个 mdnotes 实例 + 一个 SQLite 文件 + N 个 vault，每个 vault 是逻辑隔离的笔记集合。

**架构影响**：

- 数据库：vault 表（id / name / path / created_at）
- notes 表新增 `vault_id FK`
- CLI 新增 `--vault <name>` 全局选项（类似 `git -C <repo>` 但是运行时切换）
- 默认 vault = `default`

**关键约束**：

- vault 切换**不换数据库文件**（单 SQLite，多 vault 是逻辑层隔离）
- vault 之间**不跨库 join**（解耦优先）
- migration 需要把现有 notes 迁移到 `default` vault

---

### C2 — 远程同步（云端备份 + 多端同步）

**问题**：当前笔记只在本地，用户换电脑 / 重装系统有数据丢失风险。

**目标**：支持加密上传到云存储，支持多设备同步（冲突检测 + 合并策略）。

**架构影响**：

- 新增 sync engine 模块（抽象 storage backend）
- storage backend 接口：put / get / list / delete（支持 S3 / R2 / 未来扩展）
- 加密：AES-256-GCM，内容不可读地上传
- 冲突处理：类似 ADR-0008 restore 冲突策略（overwrite / keep / manual merge）
- sync log 表（last_sync_at / remote_rev / conflict_state）

**关键约束**：

- MVP 只做备份（单向 push），双向 sync 放 v2.1
- 冲突处理 MVP = "最后一次 push 赢"，后续版本再支持 interactive merge
- 不做文件内容 diff（只做 metadata 级别的增量 sync）

---

### C3 — Web UI（REST API + 静态前端）

**问题**：当前只有 CLI，非技术用户使用门槛高；无法在手机 / 平板上查看笔记。

**目标**：FastAPI 提供 REST API，前端用静态 HTML/JS 托管，实现浏览器内查看 / 搜索 / 编辑。

**架构影响**：

- 新增 `mdnotes web` 命令（启动 FastAPI 服务）
- REST API：`/api/notes` / `/api/vaults` / `/api/search`
- 静态文件：`svelte` 或 `react` SPA（CDN 托管，不进 pip）
- Web 服务默认 localhost only，production 需手动加 `--host 0.0.0.0`
- auth：MVP 无 auth（本地使用）；v2.1 加 token-based auth

**关键约束**：

- API 先行，Web UI 可以独立迭代（先有 API 再有 UI）
- Web UI 源码不进本 repo（独立 frontend repo 或 monorepo）
- FastAPI 服务和 CLI 共享同一个 SQLite，无额外 broker

---

### C4 — Plugin 系统（可扩展的 command / renderer / storage backend）

**问题**：当前所有命令（add / delete / search 等）和渲染器（mistune）、存储后端（SQLite）都是硬编码。新增一个 renderer 或 storage backend 需要改 core 代码。

**目标**：plugin 接口注册体系，支持第三方或自研插件扩展。

**架构影响**：

- plugin manifest（`~/.mdnotes/plugins.toml` 或 `pyproject.toml` entry_points）
- plugin 接口：Command / Renderer / StorageBackend / SyncBackend
- plugin 加载：Python `entry_points` 扫描 + `pluggy` 框架
- 内置插件：default commands / mistune renderer / SQLite storage
- v2.0 MVP plugin scope = Renderer 插件化（其他三个放 v2.1+）

**关键约束**：

- plugin API 稳定性：v2.0 内核接口冻结，plugin breaking change 放 v3.0
- plugin 安全：MVP 不做 sandbox，信任本地用户
- plugin 分发：支持 pip install 方式（`mdnotes-plugin-xxx`）

## 优先级排序

### 推荐顺序： C1 → C2 → C3 → C4

**理由**：

| 优先级 | 特性 | 理由 |
|---|---|---|
| P0 = C1 | **多 vault** | 纯内部重构，破坏性最小；为 C2/C3 提供逻辑隔离基础；用户价值立竿见影 |
| P1 = C2 | **远程同步** | 数据安全是强需求；依赖 C1 的 vault 模型；MVP 只做 backup 不做 full sync，风险可控 |
| P2 = C3 | **Web UI** | 依赖 C1 + C2（先有 vault 概念 + 先有 backup）；可与 C2 并行开发（API 和前端独立）|
| P3 = C4 | **Plugin 系统** | 投入产出比最低（C1~C3 完成后核心功能已齐）；plugin API 设计需要稳定，晚了做反而更安全 |

### 不建议并行的原因

C1（多 vault）是 C2/C3 的基础——vault_id FK 进 notes 表是后面所有功能的锚点。C2 和 C3 虽然可并行（C2 后端同步 / C3 前端 UI），但两者都依赖 C1，所以 C1 必须先完成。

## 风险评估

| 特性 | 技术风险 | 业务风险 | 缓解策略 |
|---|---|---|---|
| C1 多 vault | SQLite schema migration 复杂度（新增 vault FK）；migration 失败回滚 | 现有用户升级后 notes 自动归属 default vault，行为不变 | MVP migration 先做 backup；rolling migration 分步走 |
| C2 远程同步 | S3/R2 SDK 选型 + 加密实现复杂度；冲突合并算法未验证 | 云存储成本超预期；用户对加密 key 管理不熟悉 | MVP 先只做 backup；key 管理用 passphrase 模式降低门槛 |
| C3 Web UI | FastAPI + SPA 部署复杂度；auth 缺失（v2.0） | 用户误将 Web 服务暴露公网 | 默认 localhost；production 文档明确标注安全注意事项 |
| C4 Plugin 系统 | plugin API 稳定性（做早了接口容易变）；plugin 加载安全性 | 第三方 plugin 质量不可控；社区贡献者激励 | v2.0 MVP 只做 Renderer 插件化；其他三类放 v2.1+；接口冻结文档 |

**最高风险项**：C2 远程同步的加密 key 管理和冲突合并算法，建议 v2.0 MVP 只做单向 backup，不做双向 sync。

## 实施路径

### 方案 A（推荐）：小步快跑，4 个 minor 版本过渡

```
v1.6  ─── C1 多 vault（migration + vault CLI + vault API）
v1.7  ─── C2 远程同步 MVP（单向 backup，S3/R2 存储后端）
v1.8  ─── C3 Web UI MVP（FastAPI + 静态前端，local-only）
v2.0  ─── C4 Plugin 系统（Renderer 插件化）+ breaking change 汇总
```

**优点**：
- 每个 minor 版本都有用户价值，可独立发布
- 降低单次变更幅度，减少 CI 失败风险
- v2.0 breaking change 有充分预告（3 个 minor 版本的 changelog）

**缺点**：
- v1.6 ~ v1.8 每个版本都要做 migration，migration 复杂度累积
- plugin 系统放最后，v1.6~v1.8 的 renderer 仍是 hardcoded（但可接受）

### 方案 B（大爆炸）：直接 v2.0 一次性交付

```
v2.0  ─── C1 + C2 + C3 + C4 全部交付
```

**优点**：
- 一次 migration，v1.x → v2.0 一步到位
- plugin 系统和 vault 可协同设计（plugin 可声明支持的 vault）

**缺点**：
- 单次变更面太大，CI 失败排查困难
- C2/C3 复杂度高，一次性集成风险叠加
- 无 beta feedback 机会，v2.0 发布后才发现问题代价高

**结论：采纳方案 A**。v1.x → v1.6 → v1.7 → v1.8 → v2.0 的演进路径，每步都有交付价值，风险可控。

## v1.x → v2.0 Breaking Change 预告

v2.0 预计包含以下 breaking change（具体范围在 v1.8 后汇总确认）：

| Breaking Change | 影响 | 迁移路径 |
|---|---|---|
| notes.vault_id FK | 现有 notes 迁移到 default vault | 自动 migration，CLI 提示"迁移 N 条笔记到 default vault" |
| `mdnotes --vault` 全局选项 | 旧脚本带其他 `--vault` 参数会报错 | v1.9 release note 提前预告；`--vault` 默认值 default 兼容旧行为 |
| CLI 参数标准化（可选） | 若统一子命令结构，某些 alias 可能废弃 | v1.9 前保留 alias，v2.0 changelog 说明 |

**非 breaking**（v2.0 保持兼容）：
- SQLite 文件路径不变（`~/.mdnotes.db`）
- 现有 CLI 命令（add / list / show / delete / search 等）参数不变
- v1.x 的 `mdnotes.db` 可被 v2.0 直接打开（migration 自动执行）

## v1.6 ~ v2.0 各版本验收标准

| 版本 | 核心交付 | Done 标准 |
|---|---|---|
| **v1.6** | 多 vault（migration + CLI + 查询）| `mdnotes vault create work` / `mdnotes --vault work add` / `mdnotes vault list` 可用；旧 notes 在 default vault；211 测试全部通过 |
| **v1.7** | 远程同步 MVP（单向 backup）| `mdnotes sync push --provider r2` 可上传加密备份；`sync pull` 可拉回；无冲突检测（MVP）|
| **v1.8** | Web UI MVP（local-only）| `mdnotes web` 启动 FastAPI；浏览器可查看 / 搜索 / 增删改笔记；无 auth |
| **v2.0** | Plugin 系统（Renderer 插件化）+ breaking change 汇总 | `pip install mdnotes-plugin-html` 可注册新 renderer；v1.x → v2.0 migration guide 完整；changelog 列出所有 breaking change |

## 结论

v2.0 是架构升级，不是功能堆砌。推荐路径：

1. **v1.6**：多 vault（逻辑隔离，为所有后续功能打基础）
2. **v1.7**：远程同步 MVP（单向 backup，数据安全）
3. **v1.8**：Web UI MVP（降低用户门槛，API 先于 UI）
4. **v2.0**：Plugin 系统（Renderer 插件化）+ breaking change 汇总发布

4 个 minor 版本过渡，每个版本独立可发布，降低单次变更风险，v2.0 正式版有充分的用户反馈基础。
