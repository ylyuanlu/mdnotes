## 4 个教训

### 教训 1：v1.5 breaking change 导致 CI 一次失败（已修复）

**现象**：`delete` 行为从物理删除变更为软删除（默认），导致依赖旧行为（物理删除）的 4 个 v1.0 旧测试失败。

**根因**：v1.5 引入 breaking change（ADR-0008 §后果：负面 3），但 QA 没有在 coding → reviewing 阶段提前跑 integration test 覆盖边界场景。

**处置**：QA 在 testing 阶段修复 4 个旧测试（`test_delete_*` 边界条件），commit `21ce978`。

**教训**：breaking change 引入时，reviewer 和 QA 需要特别关注"旧行为依赖"的测试覆盖。

### 教训 2：5 处状态不一致部分修复（archive 字段缺失）

**现象**：v1.0 遗留 5 处状态不一致（`task.json` 的 `status` 字段与实际 agent 阶段不符），其中 `archive` 字段在 v0.1 归档任务中缺失。

**根因**：早期 task 未按 v0.5 规范写 `milestones[]` + `archived_at`，历史数据未补全。

**处置**：主公已清理，剩余 `archive` 字段规范写入 `openclaw-sdlc-framework-v0.1.md`。

**教训**：状态机规范需要持续审计，老任务的状态补全是技术债。

### 教训 3：本地 integration test 路径问题已加 conftest.py

**现象**：本地跑 integration test 时路径解析失败（conftest.py 未注册 `tests/` 为 package）。

**根因**：`__init__.py` 缺失，`pytest` 未识别为 package。

**处置**：QA 加 `conftest.py` 并确保 `__init__.py` 存在，本地路径问题修复。

**教训**：测试基础设施（conftest / `__init__.py`）应在 CI 首次失败后优先排查。

### 教训 4：CI 失败邮件无主动通知方案 A 暂缓

**现象**：GitHub CI 成功时不发邮件，CI 失败时邮件通知也不稳定。团队无法主动感知 CI 状态。

**根因**：GitHub Actions 设计如此，需要外部 webhook（飞书 / Slack）才能主动通知。

**处置**：方案 A（飞书 webhook 通知 CI 状态）因 v1.5 时间窗口有限暂缓，纳入 v1.6 或 v1.7 继续。

**教训**：CI 可观测性是工程效率基础设施，应在项目早期（v0.1）就配好。
